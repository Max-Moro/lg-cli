# Document Profile Naming Conventions

## Problem

Profile names for similar concepts differ across languages without documented rationale.

| Language | Constants/Variables | Class Fields |
|----------|---------------------|--------------|
| Python | `variable` | — |
| TypeScript | `variable` | `field` |
| Go | `constant`, `variable` | `field` |
| Rust | `const`, `static` | `field` |
| Kotlin | `property` | `property` |

## Task

Add naming convention section to `lg-cfg/adapters/code_optimization_architecture.md`:

```markdown
### Profile Naming Conventions

#### Standard Names (use when applicable)

| Name | Meaning | Languages |
|------|---------|-----------|
| `class` | Class definition | All OOP languages |
| `function` | Top-level function | All |
| `method` | Function inside class/object | All OOP languages |
| `field` | Class/struct member variable | TS, JS, Java, C++, Go, Rust |
| `variable` | Module/package-level variable | Python, TS, JS, Go |
| `interface` | Interface definition | TS, Java, Go |
| `enum` | Enum definition | TS, Java, Kotlin, Rust, C++ |

#### Language-Specific Names (preserve language semantics)

| Name | Language | Reason |
|------|----------|--------|
| `property` | Kotlin | Kotlin's val/var are properties with getters/setters |
| `const`, `static` | Rust | Rust distinguishes const (compile-time) from static (runtime) |
| `constant` | Go | Go has explicit const declarations |
| `trait` | Rust, Scala | Language-specific abstraction mechanism |
| `object` | Kotlin, Scala | Singleton pattern built into language |
| `case_class` | Scala | Distinct from regular class |

#### Rules

1. **Preserve language semantics** — if language distinguishes concepts, use separate profiles
2. **Use standard names** — when language concept maps directly to standard
3. **Don't force unification** — `property` in Kotlin is NOT the same as `field` in Java
```
